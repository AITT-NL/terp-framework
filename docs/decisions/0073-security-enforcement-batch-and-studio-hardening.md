## Context

A usability-and-enforcement review asked what an agent (or a rushed human) could
still ship that the gate would wave through, and what the local Studio trusted
that it should not:

- **Injection surfaces.** A dynamically built `text(f"...")` statement is
  un-reviewable SQL injection; `dangerouslySetInnerHTML`, the DOM HTML-injection
  sinks, `eval()` and `javascript:` URLs are the frontend equivalents. None was
  refused.
- **Ungoverned egress and secrets.** A module could import `httpx` and call
  anywhere (no SSRF/allowlist/timeout policy), hard-code an API key, or open a
  `WebSocket` past the generated-client seam.
- **Silent data loss.** An autogenerated migration that drops a table or column
  applied without any human-visible marker.
- **The Studio's blind trust.** The local Studio API had no browser boundary
  (DNS rebinding + drive-by cross-site POSTs reached `workspace/discard`),
  accepted commits with a red or stale gate, would happily bind non-loopback,
  and stored its state in JSON files a crash could corrupt.

## Decision

### 1. Four backend arch rules

Joining `_ALL_RULES` (and therefore the harness self-completeness meta-test and
the generated `terp guide rules` surface, ADR 0030):

- **`no_dynamic_sql`** — the argument to `text(...)` must be a static literal;
  data flows through bound parameters, never f-strings / concatenation /
  `.format` / `%`.
- **`no_raw_outbound_http`** — app modules must not import `httpx` / `requests`
  / `urllib.request` / `urllib3` / `aiohttp`; outbound HTTP belongs behind a
  declared capability that centralizes SSRF protection, allowlists, egress
  auditing and timeout policy.
- **`no_hardcoded_credentials`** — a credential-shaped assignment to a string
  literal, or a recognizable secret-token literal anywhere in a module, is
  refused; secrets come from settings / sealed config (ADR 0055).
- **`no_destructive_migrations`** — destructive DDL (drop table/column,
  alter-column type change) in a revision is refused unless the revision carries
  a justified `# terp-allow-destructive-migration: <reason>` marker, making data
  loss a visible, reasoned decision.

### 2. Frontend security defaults (strict-only, per ADR 0059)

Four named rules join `@terp/eslint-boundaries`, each an error with the same
governed `// terp-allow-<rule>: <reason>` + budget opt-out:

- **`no-dom-html-injection`** — `innerHTML` / `outerHTML` / `insertAdjacentHTML`
  / `document.write` are refused (with `dangerouslySetInnerHTML` already refused
  in app modules); rich text renders through react-core's dependency-free, safe
  `Markdown` component, which never passes raw HTML through.
- **`no-eval`** — `eval()` and `new Function()` are refused.
- **`no-unsafe-href`** — `javascript:` URLs in static `href`/`src` are refused.
- **`no-unsafe-target-blank`** — a static `target="_blank"` link requires
  `rel="noopener"`.

The generated-client seam is also completed: `WebSocket` / `EventSource` /
`navigator.sendBeacon` constructors and calls are refused alongside raw
`fetch` / `XMLHttpRequest`, so the typed client is the single egress path.

### 3. Studio security posture (local mode)

The local Studio keeps no login but stops trusting the browser next to it:

- **Browser boundary** — fail-closed (403) middleware: every request's `Host`
  must name loopback (or a `STUDIO_TRUSTED_HOSTS` entry), defeating DNS
  rebinding; every mutating request with an `Origin` header must match the same
  allowlist, defeating drive-by cross-site POSTs.
- **Gate-enforced accept** — `POST /workspace/commit` refuses (409) unless the
  last gate run is green *and* current for the checkout; `?force=true` is an
  explicit human override, never a default. Edits to gate-defining files
  (`pyproject.toml`, `package.json`, eslint configs, the escape-hatch budget,
  `.github/`, `conftest.py`) are flagged as **gate config** in review, since an
  agent editing those can turn red green without fixing anything.
- **Loopback bind is enforced** — `python -m terp_studio` refuses a non-loopback
  `STUDIO_HOST` without `STUDIO_ALLOW_REMOTE=true`.
- **Crash-safe state** — every Studio JSON store writes atomically, quarantines
  a corrupt file instead of failing every request, and serialises concurrent
  writers; the app supervisor persists started stacks so a crashed Studio reaps
  its orphans at the next boot.

## Consequences

- The four arch rules are projected into `terp guide rules` automatically; the
  `service` and `capability` guide topics, both `AGENTS.md` surfaces, the
  copilot instructions and the react-core README now teach the same rules, so
  the docs-parity gate keeps the claims honest.
- The example app's budgets stay `{}` — nothing in the maintained surface needed
  an escape hatch.
- The Studio posture is documented in `studio/README.md` (Security posture
  section) and covered by the Studio test suite; it is deliberately *not* an
  auth layer — exposing the API remotely remains a decision behind an
  authenticating proxy.
