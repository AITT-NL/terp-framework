# ADR 0056 ‚Äî `terp-cap-files`: owner-scoped file objects on a pluggable storage backend

- Status: Accepted
- Date: 2026-07-02
- Phase: Phase 2 capabilities (design ¬ß3.1, ¬ß6 ‚Äî "files (pluggable storage)", the last
  unbuilt capability alongside `projects`)
- Relates to: ADR 0051 (webhooks ‚Äî the sibling capability this mirrors for structure: an
  owner-scoped table + admin entry-point router + `terp.migrations` history + a sensitive
  field that never leaves the boundary), ADR 0046/0048 (the engine-adapter pattern the
  storage seam applies: a tiny port + a shipped reference adapter + one-line swapping), ADR
  0029 (`OwnedMixin` ‚Äî the per-row write gate on files), ADR 0020 / 0037 / 0040
  (`response_model_not_table_model` + `schemas_exclude_sensitive_fields` ‚Äî the storage key
  never leaves the boundary), ADR 0027 (packaged per-capability migrations), ADR 0006 (the
  two-layer runtime + build-time discipline)
- Adds **no** new `terp.core` surface and **no** storage-engine SDK: it is built entirely
  on ports that already shipped, so the vendored core mirror is untouched.

## Context

The design's capability roster (¬ß3.1, ¬ß6) has one foundation capability left unbuilt:
**files** ‚Äî let an app store and retrieve file objects through a maintained,
secure-by-default surface instead of hand-rolling uploads. Two hard problems recur in every
hand-built version: *where the bytes live* (a deployment concern ‚Äî local disk in the
workbench, object storage in production ‚Äî that must never leak into app code), and *who may
touch a file* (the per-row authorization the platform already solves centrally). The
platform has seams for both: the engine-adapter pattern (ADR 0046/0048) for the byte store,
and `OwnedMixin` + `BaseService` (ADR 0029) for ownership.

## Decision

### 1. `terp-cap-files` ‚Äî one metadata table, one entry-point router

An opt-in capability (discovered at `/api/v1/files`, like `terp-cap-webhooks`) owning one
table with an independent Alembic history (`alembic_version_files`):

- **`File`** (`BaseTable` + `OwnedMixin`, table `file_object`): `filename`,
  `content_type`, `size`, `sha256`, and the server-generated `storage_key` ‚Äî every
  caller-influenceable `str` length-capped. Composing `OwnedMixin` makes the per-row write
  gate authorize rename / delete to the owning admin with **no** module code; the router is
  `ADMIN`-only (file storage is a privileged, disk/backend-consuming surface).
- **Mutability is decided per field**: `filename` is mutable (a rename is a metadata edit,
  OCC-guarded via the required `version`); `content_type` / `size` / `sha256` /
  `storage_key` are **append-only** ‚Äî derived from the uploaded bytes at create time and
  never patched (`FileUpdate` carries no field for them; replacing content is a new
  upload). This keeps the digest / size trustworthy: they always describe the stored bytes.

### 2. The storage seam: a port + a shipped local adapter (the ADR 0046/0048 pattern)

The **metadata is owned in the platform database**; the **bytes flow through a port**:

- `StorageBackend` ‚Äî an ABC with exactly `put` / `get` / `delete`. `get` raises
  `FileNotFoundError` for an unknown key (mapped to a typed 404); `delete` is
  **idempotent** (an unknown key is a no-op), so compensations and replays never fail on
  already-gone bytes.
- `LocalFilesystemStorage` ‚Äî the default reference adapter (blobs under one root, default
  `./var/files`). Keys are server-generated (opaque UUID-derived, two-level sharded), and
  the adapter still fail-closes on any key that would resolve outside its root ‚Äî defense in
  depth against a future adapter / key-shape change reintroducing traversal.
- The active backend is installed through `set_storage_backend` /
  `active_storage_backend` / `reset_storage_backend` (the webhook-sender seam shape): an
  S3-style adapter is another `StorageBackend` subclass swapped in with one line ‚Äî no
  composition-root shape change, no service / router / schema edit.

**Transactional posture:** the upload writes the blob first (under a fresh key), then
creates the metadata row through the audited `BaseService` chokepoint; a failed row write
**compensates the blob away**, so a failed upload leaves nothing behind and a committed row
always has its bytes. The delete runs the owner-gated, audited row delete **first** (the
authorization decision), then removes the blob ‚Äî a caller who may not delete the row can
never destroy its bytes.

### 3. Sensitive-field posture: the storage key never leaves the boundary

`storage_key` is server-side-only material (the raw storage address). It is generated by
the service (never accepted from a client ‚Äî `FileCreate` is constructed server-side from
the multipart upload, never parsed from a client JSON body), and **no Read DTO carries
it**: `FileRead` omits the field, the `schemas_exclude_sensitive_fields` posture governs
the schemas, and a runtime test asserts a validated, serialized `FileRead` contains
neither the field nor the value ‚Äî the same two-layer treatment as the webhook `secret`.

The download route returns the bytes through an explicit `Response` (stored media type +
a sanitized RFC 5987 `Content-Disposition`, so a hostile stored filename cannot inject
response headers). It is the capability's one governed escape hatch: a binary route cannot
declare a `response_model`, so the `routes_declare_response_model` opt-out carries a
justified `# arch-allow-*` marker ratcheted by `escape-hatch-budget.json` (count: 1).
Uploads are size-capped at the boundary (25 MiB) before any byte is buffered or stored.

## Consequences

- The base-profile OpenAPI contract and the example app's spec gain the `/api/v1/files`
  surface (both committed artifacts + their generated TypeScript schemas regenerated).
- `terp migrate upgrade` now applies a `files` history for any app with the capability
  installed; the conformance gate covers upgrade ‚Üí downgrade for the `file_object` table.
- The capability is arch-scanned like every other (`test_capability_arch`), budgeted at
  exactly one opt-out, and held to the 100% framework coverage gate.
- Deferred: a streaming upload/download path (today's surface buffers within the 25 MiB
  cap), an S3/Azure-blob adapter package, per-file visibility beyond the admin+owner
  posture, and content-addressed dedup.
